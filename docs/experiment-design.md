# Generative Specification Controlled Experiment
## Design Document & Pre-Registration

**Version:** 1.0  
**Date:** March 11, 2026  
**Status:** Pre-registered — design committed before implementation begins  
**Commit intent:** This document is committed BEFORE either implementation is run, establishing that the evaluation criteria, expected outcomes, and fairness constraints were specified prior to observing results.

---

## 1. Research Question

Does a full Generative Specification artifact cascade (CLAUDE.md, ADRs, C4 diagrams, use cases, schema definitions, commit hooks, test architecture, MCP config) produce structurally superior output compared to baseline "raw prompts only" conditions, when both are given identical implementation instructions?

The hypothesis, stated falsifiably: **Treatment output will score higher than control output on at least 4 of the 6 GS properties, and will exhibit fewer layer boundary violations, higher test coverage, and higher API spec conformance.**

---

## 2. Benchmark: RealWorld (Conduit) Backend API

**Source:** https://github.com/realworld-apps/realworld  
**Why this benchmark:**
- 83k GitHub stars — well-known, community-validated
- A defined, stable API spec (ground truth for "correct" behavior)
- 100+ independent implementations across languages and frameworks (structural ground truth)
- Complex enough for architectural drift to manifest: auth, multi-resource CRUD, relationships, pagination, filtering
- Not so complex that it resists completion in a session: ~15 API endpoints, ~5 domain entities
- TypeScript/Node.js implementation is natural for ForgeCraft's primary stack
- Has a passing test suite that can be run against any implementation

**Domain entities:** User, Article, Comment, Profile, Tag  
**Features:** Authentication (JWT), user management, article CRUD, comment CRUD, profile follow/unfollow, tag listing, feed, favoriting  
**Stack (both conditions):** TypeScript, Node.js, Express, Prisma, PostgreSQL, Jest

## 3. Conditions

### 3.1 Control (`experiments/control/`)

**What the agent receives:**
- The RealWorld API spec (OpenAPI/markdown, the external ground truth)
- A brief problem statement (see §5.1)
- The implementation prompts (aligned in feature content with treatment — see §6 for differences)

**What the agent does NOT receive:**
- Any CLAUDE.md or architectural guidance
- ADRs
- Diagrams
- ForgeCraft output of any kind
- The GS paper or any description of the GS methodology
- Any mention of layers, SOLID, TDD, or clean architecture

**Intent:** Simulate the common practitioner baseline — "here is the spec, build it" — with no architectural scaffolding.

### 3.2 Treatment (`experiments/treatment/`)

**What the agent receives:**
- Same RealWorld API spec
- Same problem statement
- **ForgeCraft-generated artifact cascade** (see §4)
- Same implementation prompts (aligned feature content — see §6 for documented differences in test timing and commit gate)

**What the agent does NOT receive:**
- The GS paper (the artifact cascade is the specification; the paper is not injected into context)

**Intent:** Simulate GS-disciplined development — architecture first, then implementation — using ForgeCraft to generate the artifact cascade.

### 4.3 Principle: No Overfit

ForgeCraft templates are **not modified** for this experiment. Any template change that would benefit this experiment but not the general case is prohibited. The treatment's structural advantage must come from applying existing ForgeCraft capabilities correctly, not from tuning the tool to the benchmark.

## 5. Problem Statements

### 5.1 Control Problem Statement

```
Build a backend REST API for a social blogging platform called Conduit.

The API spec is attached. Implement it completely in TypeScript using Node.js, Express, Prisma, and PostgreSQL.

Requirements:
- All endpoints in the attached spec must work correctly
- Users can register, login, and manage their profile
- Users can create, read, update, and delete articles
- Users can comment on articles
- Users can follow other users
- Users can favorite articles
- Tags can be listed
- The feed endpoint returns articles from followed users
```

### 5.2 Treatment Problem Statement

```
Build a backend REST API for a social blogging platform called Conduit.

The API spec is attached. The architectural constitution (CLAUDE.md), ADRs, diagrams, and schema have been prepared. Implement it completely in TypeScript using Node.js, Express, Prisma, and PostgreSQL, following all rules in CLAUDE.md.

Requirements:
- All endpoints in the attached spec must work correctly
- Users can register, login, and manage their profile
- Users can create, read, update, and delete articles
- Users can comment on articles
- Users can follow other users
- Users can favorite articles
- Tags can be listed
- The feed endpoint returns articles from followed users
```

## 6. Implementation Prompts

Issued in order after the problem statement.

**Amendment (2026-03-12, pre-implementation):** Control prompts 01–05 do not include test instructions. A dedicated test prompt (07) is issued after all features are implemented, simulating the common practitioner pattern of build-then-test. Treatment prompts retain inline test instructions in each feature prompt. This difference was introduced to produce a more realistic control baseline — a developer without GS context would typically not write tests per-feature unless explicitly architected to do so. Both conditions receive equivalent total test instruction; the variable is integration vs. deferred.

A second intentional difference: each treatment prompt appends "Before committing: run the Verification Protocol (CLAUDE.md §Verification Protocol)." This is part of the treatment's GS commit discipline and is not a feature instruction.

Identical across both conditions: feature requirements, endpoint list, error format spec, and the hardening prompt (06). Each prompt is self-contained; the agent commits after completing each.

**Prompt 1 — Authentication**
```
Implement user authentication:
- POST /api/users (register)
- POST /api/users/login (login)
- GET /api/user (get current user, auth required)
- PUT /api/user (update user, auth required)

Include input validation, JWT token generation/verification, and password hashing.
Write unit tests for auth logic and integration tests for all four endpoints.
```

**Prompt 2 — Profiles**
```
Implement user profiles:
- GET /api/profiles/:username
- POST /api/profiles/:username/follow (auth required)
- DELETE /api/profiles/:username/follow (auth required)

Write tests for all three endpoints.
```

**Prompt 3 — Articles**
```
Implement articles:
- GET /api/articles (list, with filters: tag, author, favorited; pagination: limit, offset)
- GET /api/articles/feed (auth required, articles from followed users)
- GET /api/articles/:slug
- POST /api/articles (auth required)
- PUT /api/articles/:slug (auth required, author only)
- DELETE /api/articles/:slug (auth required, author only)
- POST /api/articles/:slug/favorite (auth required)
- DELETE /api/articles/:slug/favorite (auth required)

Write tests for all endpoints including authorization checks, pagination, and filtering.
```

**Prompt 4 — Comments**
```
Implement comments:
- GET /api/articles/:slug/comments
- POST /api/articles/:slug/comments (auth required)
- DELETE /api/articles/:slug/comments/:id (auth required, author only)

Write tests for all endpoints.
```

**Prompt 5 — Tags**
```
Implement tags:
- GET /api/tags

This should return all unique tags that appear on any article.
Write a test for this endpoint.
```

**Prompt 6 — Integration & Hardening**
```
Now:
1. Run all tests and fix any failures
2. Check that every route handler delegates to a service (no direct DB calls in routes)
3. Verify error responses conform to the API spec format
4. Add any missing test coverage for edge cases and error paths
5. Produce a summary of: test count, any layer violations found and fixed, coverage %
```

## 7. Evaluation Protocol

### 7.1 Auditor Conditions

The auditor session is a **fresh context window** — no prior conversation, no knowledge of which condition produced which output. The auditor receives:
- The 6 GS properties (Self-describing, Bounded, Verifiable, Defended, Auditable, Composable) with their §4.3 definitions
- The output of one condition (code + all artifacts)
- Instructions to score each property 0–2 and provide evidence for each score

The auditor does **not** receive:
- The GS paper
- Knowledge that this is an experiment
- The other condition's output

Both conditions are audited in separate sessions. The auditor prompt is identical for both.

### 7.2 Scoring Rubric

**Each property scored 0–2:**
- 0 = Absent or architecturally violated
- 1 = Partially present (some evidence but incomplete)
- 2 = Structurally present and enforced

**Property evidence criteria:**

| Property | Evidence of 2 (Fully Present) |
|---|---|
| Self-describing | CLAUDE.md or equivalent exists, covers architecture + conventions + decisions. A stateless reader can determine what the system is from artifacts alone. |
| Bounded | Route handlers delegate to services; services delegate to repositories; no cross-layer imports. Max function length respected. |
| Verifiable | Tests present, organized by layer, written against interfaces not implementations. Coverage ≥ 80%. Test names are specifications. |
| Defended | Commit hooks present and enforcing. Lint/format blocking. TDD gate present or equivalent. |
| Auditable | Conventional commits present. ADRs present. Status.md or equivalent. Decision history recoverable from artifacts. |
| Composable | Services depend on interfaces (or equivalent abstraction). No implicit global state. Repository pattern in use. |

**Objective metrics (measured, not estimated):**

| Metric | How Measured |
|---|---|
| Test count | `npx jest --listTests` + count |
| Test coverage % | `npx jest --coverage` |
| Layer violations | Grep for `prisma.` in route handler files |
| API spec conformance | Run RealWorld API spec test suite (if available) |
| Naming signal | Manual sample: 10 random function names, scored for domain term presence |
| Commit quality | All commits use conventional format? (0/1) |
| ADR count | Count files in `docs/adrs/` |
| Has CLAUDE.md | Boolean |

### 7.3 Expected Outcomes (Pre-registered)

The following outcomes are predicted BEFORE running the experiment. Recording them before observation makes the results falsifiable.

**Predicted differentials (Treatment > Control):**

| Metric | Predicted Direction | Confidence |
|---|---|---|
| Self-describing score | T > C (by ≥ 1 point) | High — Control has no CLAUDE.md |
| Bounded score | T > C (by ≥ 0.5 point) | Medium — Layer discipline depends on CLAUDE.md rules |
| Verifiable score | T ≥ C | Medium — Both will have tests; treatment has explicit coverage gate |
| Defended score | T >> C | High — Control has no hooks by definition |
| Auditable score | T >> C | High — Control has no ADRs; may have non-conventional commits |
| Composable score | T ≥ C | Medium — Compositional structure may emerge in both |
| Layer violations | T < C | Medium-High — CLAUDE.md explicitly forbids direct DB from routes |
| Test coverage | T ≥ C | Low-Medium — Test quality depends heavily on prompt execution |
| Naming signal | T > C | Medium — Naming conventions in CLAUDE.md guide token choices |

**Predicted non-differential (null hypothesis):**
- Total LoC: expected similar
- Feature completeness (API spec coverage): expected similar — the prompts define the features

**Falsification condition:** If Treatment and Control produce identical or near-identical scores on Self-describing and Defended (properties that are definitionally absent in the Control), the GS claim fails.

## 8. What This Experiment Does Not Prove

Even if the expected outcomes are observed:
- It does not prove GS produces better outcomes for all practitioners (single-agent execution)
- It does not prove the structural improvements are attributable to GS rather than to better prompting
- It does not prove all 6 properties are independently necessary (they may be redundant)
- It does not prove the methodology scales to larger systems (RealWorld is ~500 LoC spec)

These limitations are honest and consistent with §8.8 of the paper. This is a first controlled demonstration, not a controlled trial.

---

## 9. Anti-Overfit Controls

1. ForgeCraft template files are **version-locked to the commit immediately preceding this design document**. No template changes between design commit and results commit.
2. Implementation prompts have identical feature requirements for both conditions (§6 above). Two intentional differences exist and are documented pre-implementation: (a) test timing — control defers tests to prompt 07, treatment writes inline; (b) commit gate — treatment appends the Verification Protocol to each prompt, control does not. Both differences are pre-registered in §6.
3. The auditor scoring rubric (§7.2) is derived from the paper's §4.3, not from the expected outputs.
4. If any ForgeCraft improvements are made based on benchmark study, they must be genuinely universal (benefit all `[API]` projects) and documented as a separate commit with rationale before the experiment runs.
