# ADR-0012: Post-Experiment Method Adoptions — Model Tiering, YAML Frontmatter, Normative Keywords, the Rubric Guide

**Status**: Accepted
**Date**: 2026-06-16
**Author**: Juan Carlos Ghiringhelli
**Builds on**: ADR-0011 (generative-execution gate + sentinel/EG vocabulary)

---

## Context

Since ADR-0011, two pilot experiments (**MX**, model cost & tiering; **RND-1**, specification / verification / bounded-context under pressure) and the associated method work produced concrete, *measured* guidance that ForgeCraft — the executor arm of GS — should adopt. The canonical sources are public in the GS research repo (`github.com/jghiringhelli/generative-specification`): the Compendium (§7.8.H–I, §4.1.a, §4.4), the **Rubric Scoring Guide** (`docs/white-paper/GS_Rubric_ScoringGuide.md`), and the reproducible experiments (`experiments/mx/`, `experiments/rnd-1/`). A third, **field** study — **VairixDX** (`github.com/grodriguez1983/vairixdx`), a real LLM executive-generation pipeline built and maintained under a configured GS harness — landed later with its own brownfield A/B/C cost experiment and a repeated audit-RUN log; its adoptions are recorded in §6. This ADR records what ForgeCraft adopts and the concrete implementation each adoption implies. It does not revise ADR-0011; it extends it.

## Decision

### 1. Model tiering — different models for different tasks (measured: MX)
**Default code generation to the mid-tier model (Sonnet-class); reserve the strong model (Opus-class) for planning and for escalating tasks the mid model fails.** MX measured both at 149/149 on the full RealWorld/Conduit backend with the mid model at **≈6× lower cost**, and found multi-model *tiering* unjustified when the mid model one-shots the task (the strong-model planner alone cost more than the mid model's entire build). **ForgeCraft action:** any orchestration/subagent path selects the mid tier for generation by default; the strong tier is reserved for planning and hard escalation; tiering machinery is engaged only for tasks beyond the mid model's one-shot capacity (large/complex specs). Source: Compendium §7.8.H, `experiments/mx/`.

### 2. YAML frontmatter on generated harness documents and sentinel nodes
Adopt the **closed-key** YAML frontmatter schema (Self-describing made machine-readable): `id / type / status / tier / properties / obligations / generative_execution / depends_on`, with a `sentinel-node` specialization carrying `node / scope / load / categories / routes_to`. **ForgeCraft action — this is the consumer that makes the schema live:** (a) the **sentinel renderer emits** the frontmatter on the docs and sentinel nodes it generates; (b) a **well-formedness gate validates** the sentinel tree — root within the bounded line budget and the only `load: always` node, the five required categories collectively present (fail a tree with no `tool-sequencing` node), every `routes_to` resolves; (c) the **drift check** (PT-2) and the router/gates **consume** the keys instead of parsing prose. Minimal-sufficient: emit frontmatter only where a consumer reads it. Source: Rubric Scoring Guide (frontmatter schema), Compendium §4.4.

### 3. RFC 2119 / 8174 normative keywords in specs
Spec obligations are phrased with **MUST / MUST NOT / SHOULD / SHOULD NOT / MAY** (normative only when capitalized, per RFC 8174). **ForgeCraft action:** spec/use-case templates use them; the keyword **sets gate severity** — `MUST → blocking gate, SHOULD → warning, MAY → ungated` — and each `MUST` becomes an acceptance criterion and therefore a probe (the `obligations` frontmatter key counts them). This reuses the existing gate-severity machinery (FC-2). Source: Compendium §4.1.a.

### 4. Prescriptive over descriptive specs (measured: RND-1)
RND-1 showed a *descriptive/ambiguous* spec makes the model floor to the literal minimum (0/3 against the held-out intent) at equal token cost, while a *prescriptive* spec recovers full intent (3/3) — the **prescriptive-specification arm is the demonstrably load-bearing one** at current model capability. **ForgeCraft action:** the spec discipline and its gates favor prescriptive form — use cases with explicit postconditions and acceptance criteria, obligations phrased with §3's keywords. (Verification and bounded-context remain insurance whose value shows against weaker/adversarial agents and at large scale — RND-1 returned honest bounding nulls there.) Source: Compendium §7.8.I, `experiments/rnd-1/`.

### 5. The Rubric Scoring Guide as the scoring contract
**ForgeCraft action:** the `verify`/score path consumes the per-property **0/1/2 calibration anchors** from `GS_Rubric_ScoringGuide.md` (not ad-hoc heuristics); the `properties` frontmatter key attributes each artifact's contribution to `gs_score`; Defended scores derived without human review are flagged provisional. Source: Rubric Scoring Guide. **(LIVE — `src/analyzers/anchors/anchor-loader.ts` + `gs-scorer.ts`.)**

### 6. Field-study adoptions (VairixDX brownfield replication)
A third study landed after §1–§5: **VairixDX**, a real LLM *executive-generation* pipeline built and maintained under a configured GS harness, with its own A/B/C cost experiment and a repeated audit-RUN log. It both **validated** the thesis (see ADR-0011 Consequences: context-precision the mechanism; hooks a ~63% token multiplier; rework dominates, naive 3.06×) and surfaced concrete harness techniques ForgeCraft does not yet generate. Each is recorded with status; all are PLANNED unless marked.

- **a. Targeted spec loading — spec-as-sections + a generated spec-map + router. _(LIVE)_** Keep the spec as `docs/specs/sections/*.md` with a `SPEC-INDEX` router and a small `spec-map` that points each requirement/use-case at exact spec line ranges. Measured on VairixDX: **~82% fewer spec tokens per task** *and* better attention (relevant content stops landing in the degraded zone past ~2k lines). This is the single largest efficiency lever observed and directly serves the Bounded/§2 `routes_to` theme. **Shipped:** (1) the sentinel renderer emits `.claude/spec-map.md` — a tag-aware "Working on → read first" cheat-sheet routing API/UI/AI-pipeline/seed/test-case tasks to the owning spec slice, with the *load-the-slice-not-the-spec* directive, wired into the root Navigate-by-Task table and the `routes/docs.md` reading order (load-on-demand, never always-load); (2) the scaffolder now emits the spec **sections** themselves — `docs/specs/sections/{api,ui,pipeline,seed,test-cases}.md` (tag-gated, prescriptive RFC 2119 stubs) plus the authoritative `docs/specs/SPEC-INDEX.md` router (`writeSpecSections` / `SPEC_SECTION_DEFS` in `src/tools/scaffold-spec-stubs.ts`). The section catalog and the spec-map routing rows share one tag predicate, so a canary asserts every advertised pointer resolves. The monolithic `docs/PRD.md` remains the fallback the SPEC-INDEX supersedes as sections fill.
- **b. Step-gated session (inversion of control). _(LIVE)_** A session manifest whose step completion is enforced by a hook: a code commit is blocked until the upstream steps are done. This is the harness *forcing the steps* rather than narrating them. **Shipped:** the scaffolder emits `.claude/session-manifest.yaml` — a flat `step: pending|done` ledger (`intake`, `spec-validation`, `red`, `green`, `refactor`, `close`) — and a `commit-msg-session-gate` hook that **blocks any `test:`/`feat:`/`fix:` commit until `intake` and `spec-validation` are `done`**, so the AI cannot start coding before it has loaded the spec slice and confirmed acceptance criteria. lifecycle.md routes the AI to the manifest at session start. *Enforcement-point note:* the RED→GREEN ordering was already enforced by the commit-msg TDD gate; this adds the steps that previously lived only in prose. It runs at `commit-msg` (the reliable interception point with the finalized message) rather than at `PreToolUse` (where a `git commit` message arrives only as opaque Bash-tool input) — the inversion-of-control property (0-context, harness-forced) holds either way; the PreToolUse layer remains for tool-level Tier-3 gating. Opt-in by manifest presence; `FORGECRAFT_SKIP_SESSION_GATE=1` is the one-off escape hatch. Covered by a 7-case bash-hook integration test (`tests/tools/session-gate.test.ts`).
- **c. Two-stream discovery log + close-gate. _(LIVE)_** Distinguish **D-XXX** (spec deviations / debt, registered *before* coding) from **DELTA-NNN** (runtime discoveries, registered *after* generative execution), and **require a captured regression fixture (the exact triggering input) to close a DELTA**. **Shipped:** the scaffolder emits `docs/discovery-log.md` (the two-stream ledger; `buildDiscoveryLog` in `src/tools/discovery-log.ts`), a comment-aware parser, and a pure evaluator `evaluateDiscoveryLog` that flags any DELTA marked `closed` whose `Fixture:` path is missing or does not resolve to a real file. `close_cycle` Step 1.8 blocks (`ready:false`, `discoveryLogStatus` on `CloseCycleResult`) on such a DELTA — a lesson that lives only in a changelog returns under a cousin input (VairixDX DELTA-079/051/007); a lesson that lives in a fixture cannot. Opt-in: the gate is skipped when no discovery log exists. Composes with the LIVE `regression-fixture-on-bugfix` gate (commit-time) and `harvest-debt` (PT-4); this is the auditable ledger + its acceptance rule.
- **d. EDR (spec-change records) with `Affected UCs:` → cascade re-verify. _(LIVE)_** A spec change records the use-cases it touches; the cycle then **re-verifies those UCs**. This composes directly with FC-1 (ADR-0011) — it is how a spec edit proves it did not silently break a downstream UC. **Shipped:** an EDR is a `docs/edrs/EDR-NNN-*.md` record carrying `Date:` + `Affected UCs:` (format guide emitted as `docs/edrs/README.md`; `buildEdrsReadme`). The pure evaluator `evaluateSpecChangeCascade` (`src/tools/spec-change-record.ts`) loads each affected UC's `lastRunAt` from `verification-state.json` and flags any UC whose generative-execution evidence is **stale** — absent, or older than the EDR date. `close_cycle` Step 1.85 blocks (`ready:false`, `specChangeCascadeStatus` on `CloseCycleResult`) on a stale UC, forcing a `run_harness` re-run after the spec change. FC-1 trusts a green; this adds the missing *staleness* axis (a green that predates the spec edit is not trustworthy). lifecycle.md's close-of-session invariant now lists the EDR step. Opt-in: skipped when no EDRs exist. *(Day-granular date comparison; a future `generate_edr` tool could replace hand-authoring.)*
- **e. "Known Pitfalls" + "Techniques" sections. _(LIVE)_** Distinct from corrections: *Known Pitfalls* = class-level bug workarounds (e.g. structured-output hangs on large prompts; a tool resolving to the wrong binary), *Techniques* = reusable patterns the project invented. Captures what tests cannot. **Shipped:** the renderer emits `.claude/pitfalls.md` with both framed sections (`buildPitfallsFile`). Placement note — these were *not* put in the always-loaded constitution/corrections as first sketched: they accumulate entries over a project's life and would breach the always-load line budget (measured: +13 lines over the 175 cap). Per this ADR's own CNT refinement (*knowledge sits behind pointers*), they live in a load-on-demand file with a pointer from the always-loaded `corrections.md`. The budget canary holds.
- **f. LLM-pipeline disciplines for the ML/AI tag. _(LIVE)_** When the generated system is itself an LLM pipeline: **separate extraction (LLM, temperature 0, structured-output JSON schema, mandatory evidence excerpt) from deterministic scoring**; add a **structural rescue** (deterministic recall floor injected *after* the LLM's selection step — "the model cannot stochastically miss what it never had to pick"); **anti-double-counting** of echoed signals; **evidence-strength levels, not LLM confidence percentages**; **replay fixtures** for LLM-output tests (the `llm-output-replay-fixture` gate is already LIVE); and the **audit-RUN multi-run scoring** discipline (ADR-0011 refinement). **Shipped:** a new `llm-pipeline-disciplines` ML instruction block (`templates/ml/instructions.yaml`, mapped to the `ml` domain in `sentinel-domain-map.ts`) emits all six disciplines (RFC 2119 phrased) into an ML project's `.claude/standards/ml.md`; and a new registry gate `stochastic-uc-run-distribution` (`.forgecraft/gates/registry/ml/`) requires a stochastic UC to be accepted on an **N-run pass-rate distribution**, not a single green. Distinct from the general model-training ML blocks — this is the *systems-built-on-an-LLM* subset. Covered by an end-to-end ML-scaffold test asserting each discipline reaches `standards/ml.md`, plus the registry-wide gate-schema ratchet.

> CNT refinement observed: VairixDX's sentinel root is ~204 lines (denser than ForgeCraft's ≤80) yet the harness measurably *helps*, because the **operating protocol stays in one auto-loaded bounded file, knowledge sits behind pointers (spec-map), and enforcement lives in hooks (0 context)**. The lesson is not "go monolith" — it refines §2's bounded-root requirement toward *operating-rules-in-root, knowledge-behind-pointers, enforcement-in-hooks*.

## What is LIVE vs PLANNED
- **Live (ADR-0011 + this session):** generative-execution gate + per-UC green/red flag (FC-1); static-analyzer gate (FC-2); multi-agent sentinel source-of-truth + drift check (PT-2); debt-marker harvest (PT-4); **`verify` consuming the Rubric Scoring Guide 0/1/2 anchors (§5)** — `loadAnchor`/`applyAnchor` in `src/analyzers/anchors/anchor-loader.ts`, consumed by `gs-scorer.ts`, with anchorless scores flagged `provisional`. **The entire §6 field-study backlog is now LIVE:** §6a targeted spec loading (`.claude/spec-map.md` + emitted `docs/specs/sections/*` + `SPEC-INDEX.md`); §6b step-gated session (`.claude/session-manifest.yaml` + `commit-msg-session-gate`); §6c two-stream discovery log + fixture-on-close gate (`close_cycle` Step 1.8); §6d EDR spec-change records + cascade re-verify (`close_cycle` Step 1.85); §6e Known-Pitfalls/Techniques behind a pointer (`.claude/pitfalls.md`); §6f LLM-pipeline disciplines for the ML tag (`llm-pipeline-disciplines` block + `stochastic-uc-run-distribution` gate). The sentinel additions are load-on-demand so the always-load budget canary still holds.
- **Planned (this ADR's implementation backlog):** sentinel-renderer frontmatter emission + the tree well-formedness gate (§2); keyword→gate-severity wiring (§3). Model-tiering (§1) and prescriptive-spec discipline (§4) are operating guidance now; their enforcement (a tiering orchestrator; a descriptive-spec linter) is optional future work. Smaller follow-ups noted inline: a `generate_edr` tool to replace hand-authoring EDRs (§6d), and run-distribution wiring into FC-1 itself for stochastic UC probes (§6f / ADR-0011 refinement).
