# ADR-0003: Adopt the Seven-Property Generative Specification (GS) Model

**Date:** 2026-03-17
**Status:** Accepted

## Status

Accepted

## Context

ForgeCraft needs a scoring model that measures whether a project's specification is complete
enough to constrain LLM code generation reliably. An incomplete specification introduces degrees
of freedom that cause generation to drift across iterations.

Five verification experiments (control, naive, treatment v1–v5) were run against a real-world
Node.js/PostgreSQL project. Each experiment tested a different specification completeness level
and measured convergence rate (iterations to stable passing state), layer violations, and
test pass rate. Results were analysed and the properties that most strongly predicted convergence
were identified.

The initial design used six properties (Self-describing, Bounded, Verifiable, Defended, Auditable,
Composable). Treatment v5 identified that passing static analysis did not guarantee passing
runtime contracts — a seventh property was needed.

## Decision

Model specification completeness as seven properties, each scored 0–2 (max 14):

1. **Self-describing** — The spec explains its own architecture, conventions, and domain.
2. **Bounded** — Layer boundaries are enforced; no cross-layer leakage.
3. **Verifiable** — Correctness checks exist, have coverage, and pass static analysis.
4. **Defended** — Quality gates block non-conforming changes from merging.
5. **Auditable** — Every non-obvious decision is recorded with rationale (ADRs, Status.md).
6. **Composable** — The system is decomposed into swappable units with clear interfaces.
7. **Executable** — The implementation actually passes its contracts at runtime (not just statically).

The pass threshold for overall PASS verdict is ≥ 11/14. The theoretical information-theoretic
model `I(S) ≈ 1/S` predicts hyperbolic convergence — each additional GS point approximately
halves the expected remaining iterations. This is deductive, not empirically calibrated, but
was qualitatively confirmed by the experiment series.

## Alternatives Considered

- **Six properties (original model)**: Did not distinguish "tests exist and pass static analysis"
  from "tests pass against real runtime". Treatment v5 showed this gap is material. Rejected.
- **Binary pass/fail per property**: Loses the signal of partial compliance (score 1). A project
  with partial coverage is meaningfully different from one with none. Rejected.
- **Continuous 0–100 score**: More granular but harder to interpret and harder to threshold.
  The 0–2 per property model maps naturally to "absent / partial / complete". Rejected.
- **Domain-specific property sets**: Different properties for API vs CLI vs ML projects.
  Added complexity without clear benefit — all seven properties are universally applicable
  (though their implementation differs by domain). Rejected.

## Consequences

- All tooling (verify, audit, scorer) operates against seven properties. Score displays as X/14.
- The `GsProperty` union type must remain stable; adding properties is a MAJOR semver bump.
- The Executable property is optional to implement on meta-artifacts (ADRs, instruction files)
  since they are not runtime implementations; the scorer measures it from the outside via
  test evidence and CI configuration.
- Domain-specific verification strategies (API, HEALTHCARE, etc.) add domain completeness on
  top of the universal baseline — the ceiling per domain strategy reflects how much of the
  remaining uncertainty it resolves.
