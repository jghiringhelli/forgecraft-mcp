# ADR-0002: Use YAML Templates as Configuration-as-Code for Standards

**Date:** 2026-03-17
**Status:** Accepted

## Status

Accepted

## Context

ForgeCraft must encode engineering standards for 24+ domain tags (API, CLI, HEALTHCARE, FINTECH,
etc.) in a way that is: (1) readable by practitioners without running code, (2) extensible by
community contributors via PRs, (3) composable — multiple tags can be combined without merge
conflicts, and (4) consumed at runtime by a TypeScript MCP server without a build step.

Standards cover instruction blocks, quality-gate hooks, verification strategies, NFR constraints,
skill files, and review checklists. These change independently and at different rates.

## Decision

Store all standards as YAML files under `templates/<tag>/` with a typed loader that validates
and merges them at runtime. Each domain's standards live in independent files
(`instructions.yaml`, `verification.yaml`, `hooks.yaml`, etc.) that the registry composer
merges in canonical order. Variables in templates are interpolated from `forgecraft.yaml`.

TypeScript types are derived from the YAML structure (not duplicated manually), and Zod schemas
validate loaded templates at startup so malformed contributions fail loudly.

## Alternatives Considered

- **TypeScript modules per tag**: Standards as `.ts` objects. Better type safety but requires
  a build step for community contributions and makes diff reviews harder. Non-engineers cannot
  read or contribute. Rejected.
- **JSON**: More verbose than YAML, no support for multi-line strings or comments.
  YAML's block scalar (`>`) is essential for readable instruction prose. Rejected.
- **Database-backed standards**: Enables versioning and A/B testing but adds a runtime
  dependency and makes offline/CI use impossible. Deferred to post-1.0.
- **Single monolithic YAML per tag**: Simpler loading but creates merge conflicts when multiple
  concerns (instructions vs verification vs hooks) change simultaneously. Rejected.

## Consequences

- Community contributions to standards are YAML PRs — reviewable without code knowledge.
- Template composition is deterministic: UNIVERSAL always applies first; domain tags layer on top.
- The loader validates templates at startup; malformed YAML causes an immediate error, not silent
  misconfiguration.
- Template variables (`{{repo_url}}`, `{{framework}}`) are resolved from `forgecraft.yaml` and
  auto-detected project context; unresolved variables surface as literal `{{...}}` strings,
  which practitioners can grep for.
