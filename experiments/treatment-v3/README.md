# Treatment-v3 Condition — Dependency Registry (AI-Maintained, Audit-Before-Add)

## Hypothesis

Treatment-v2 achieved 12/12 on the GS structural rubric but produced 9 HIGH CVEs — more than the naive condition. Root cause: `bcrypt` pulls in a native dep chain with known tar CVEs; `@typescript-eslint@^6` pulls in an old `minimatch` with CVEs. Both were chosen by the AI without auditing.

**Hypothesis:** Prescribing `docs/approved-packages.md` as a mandatory P1 artifact, with an explicit audit-before-add rule, will force the AI to reason about CVEs at selection time and choose safe alternatives (e.g., `argon2` instead of `bcrypt`). Expected outcome: 0 HIGH CVEs, GS score maintained.

## Problem Statement

Build a backend REST API for a social blogging platform called Conduit.

The API spec is in `../REALWORLD_API_SPEC.md`. The architectural constitution (`CLAUDE.md`), ADRs, diagrams, schema, and test architecture have been prepared. Implement it completely in TypeScript using Node.js, Express, Prisma, and PostgreSQL, following all rules in `CLAUDE.md`.

Requirements:
- All endpoints in the API spec must work correctly
- Users can register, login, and manage their profile
- Users can create, read, update, and delete articles
- Users can comment on articles
- Users can follow other users
- Users can favorite articles
- Tags can be listed
- The feed endpoint returns articles from followed users
